2.2. CHI TIẾT DỰ ÁN – ỨNG DỤNG QUẢN LÝ CHI TIÊU CÁ NHÂN2.2.1. Tổng quan và mục tiêu dự ánỨng dụng Quản lý Chi tiêu Cá nhân (Personal Expense Manager – PEM) là một web application hoàn chỉnh được xây dựng nhằm giải quyết bài toán thực tế: giúp người dùng theo dõi, phân tích và kiểm soát tình hình tài chính cá nhân một cách trực quan và hiệu quả. Theo khảo sát, hơn 60% người Việt Nam chưa có thói quen theo dõi chi tiêu một cách hệ thống, dẫn đến khó kiểm soát tài chính cá nhân và tiết kiệm kém hiệu quả.Dự án được xây dựng hoàn toàn trên nền tảng AWS, thể hiện khả năng tích hợp các dịch vụ đám mây vào sản phẩm thực tế và minh chứng cho kiến thức đã tích lũy trong 3 tuần học AWS Services. Mục tiêu cốt lõi của dự án:Giúp người dùng ghi chép thu nhập và chi tiêu một cách nhanh chóng, dễ dàng.Phân loại giao dịch theo danh mục để phân tích cơ cấu chi tiêu.Cung cấp báo cáo thống kê trực quan giúp người dùng hiểu rõ tình hình tài chính.Cảnh báo khi chi tiêu có nguy cơ vượt ngân sách đã đặt ra.Hỗ trợ quản lý nhiều tài khoản tài chính (tiền mặt, ngân hàng, thẻ tín dụng...). 2.2.2. Thiết kế cơ sở dữ liệu PostgreSQLCơ sở dữ liệu PostgreSQL được thiết kế theo nguyên tắc chuẩn hóa (normalization) đến dạng chuẩn 3NF (Third Normal Form), đảm bảo tính nhất quán dữ liệu và hiệu suất truy vấn tốt. Dưới đây là mô tả chi tiết các bảng chính trong hệ thống:Bảng 2.10. Cấu trúc các bảng database chi tiếtTên bảngCác cột chínhMục đíchusersid (UUID PK), cognito_sub, email, full_name, avatar_url, currency, timezone, created_atLưu thông tin người dùng, liên kết với Cognitoaccountsid (UUID PK), user_id (FK), name, type (ENUM), balance, currency, color, icon, is_activeQuản lý các tài khoản tài chính của người dùngcategoriesid (UUID PK), user_id (FK), parent_id (FK tự tham chiếu), name, type (ENUM), icon, color, is_defaultDanh mục phân loại giao dịch, hỗ trợ phân cấptransactionsid (UUID PK), user_id (FK), account_id (FK), category_id (FK), amount, type (ENUM), description, date, receipt_url, transfer_to_account_idGhi chép mọi giao dịch tài chínhbudgetsid (UUID PK), user_id (FK), category_id (FK), amount_limit, spent_amount, period_type (ENUM), start_date, end_date, is_activeThiết lập và theo dõi hạn mức ngân sáchbudget_alertsid (UUID PK), budget_id (FK), alert_type (ENUM: 80%, 100%), sent_at, email_sent_toLịch sử các email cảnh báo ngân sách đã gửireport_exportsid (UUID PK), user_id (FK), type (ENUM: PDF/EXCEL), s3_key, s3_url, expires_at, created_atLưu trữ link download báo cáo đã xuất Các index quan trọng được tạo để tối ưu hiệu suất truy vấn:CREATE INDEX idx_transactions_user_date ON transactions(user_id, date DESC) – index tổng hợp cho query danh sách giao dịch theo user và sắp xếp ngày.CREATE INDEX idx_transactions_category ON transactions(category_id, date) – phục vụ query thống kê theo danh mục.CREATE INDEX idx_budgets_active ON budgets(user_id, is_active, start_date, end_date) – tìm ngân sách đang active.Tất cả các cột FK (user_id, account_id, category_id...) đều có index mặc định để JOIN nhanh. 2.2.4. Chức năng Xác thực người dùng (Authentication)a) Đăng ký tài khoảnLuồng đăng ký tài khoản được thiết kế bảo mật và thân thiện với người dùng, tích hợp hoàn toàn với AWS Cognito:Người dùng truy cập trang đăng ký, nhập đầy đủ thông tin: họ tên, địa chỉ email, mật khẩu (tối thiểu 8 ký tự, gồm chữ hoa, chữ thường, số và ký tự đặc biệt).Hệ thống gọi AWS Cognito API (signUp) để tạo tài khoản trong User Pool. Cognito tự động gửi email OTP 6 chữ số đến địa chỉ email đã đăng ký.Người dùng nhập mã OTP để xác thực email. Cognito xác nhận mã – nếu đúng, tài khoản được kích hoạt (CONFIRMED status).Backend NestJS tự động tạo bản ghi user trong bảng users của PostgreSQL với cognito_sub làm định danh liên kết.Sau khi đăng ký thành công, hệ thống tự động đăng nhập và chuyển hướng người dùng đến trang tổng quan. Hình 2.3. Kiến trúc triển khaiToàn bộ hệ thống được triển khai trên AWS với kiến trúc production-grade, đảm bảo tính sẵn sàng cao (High Availability) và khả năng mở rộng (Scalability):a) Frontend – AWS Amplify + CloudFrontNext.js 14 app được deploy lên AWS Amplify – dịch vụ chuyên dụng cho hosting full-stack web app. Amplify tự động detect dự án Next.js và cấu hình build phù hợp:Amplify kết nối trực tiếp với GitHub repository. Mỗi khi push code lên branch main, Amplify tự động trigger build và deploy.Build process: Amplify chạy npm ci → npm run build, tạo ra Next.js standalone output.Amazon CloudFront tự động được cấu hình phía trước Amplify, đóng vai CDN phân phối nội dung từ các edge locations gần người dùng nhất, giảm độ trễ đáng kể.Custom domain (nếu có): Amplify tích hợp với Route 53 để cấu hình domain, tự động provision SSL certificate qua AWS Certificate Manager (ACM).Environment variables: Các biến môi trường như Cognito User Pool ID, API endpoint được cấu hình trực tiếp trong Amplify Console – không lưu trong code. b) Backend – AWS ECS FargateNestJS backend được containerize bằng Docker và deploy lên Amazon ECS (Elastic Container Service) với launch type Fargate:Dockerfile multi-stage: Stage 1 (builder) cài dependencies và build TypeScript → Stage 2 (production) chỉ copy dist/ và node_modules production, giảm image size.Docker image được push lên Amazon ECR (Elastic Container Registry) với tag theo commit hash (git SHA).ECS Task Definition: Cấu hình container specs (0.5 vCPU, 1GB RAM cho môi trường dev), environment variables từ AWS Secrets Manager và Parameter Store.ECS Service: Đảm bảo luôn có ít nhất 1 task đang chạy, tích hợp với Application Load Balancer (ALB) để nhận traffic từ internet.Auto-scaling: Cấu hình ECS Application Auto Scaling dựa trên CPU utilization – tự động scale-out khi CPU > 70%, scale-in khi CPU < 30%.Rolling deployment: Khi deploy phiên bản mới, ECS dùng rolling update để đảm bảo zero-downtime. c) Database – Amazon RDS PostgreSQLPostgreSQL database chạy trên Amazon RDS với cấu hình production-ready:Instance type: db.t3.micro (2 vCPU, 1GB RAM) cho môi trường dev/staging; db.t3.small cho production.Storage: 20GB SSD (gp2), với Storage Autoscaling bật – tự động tăng khi sử dụng > 90%.Multi-AZ Deployment: RDS tự động duy trì standby replica ở Availability Zone khác, tự động failover trong < 60 giây nếu primary instance gặp sự cố.Backup: Automated backup hàng ngày lúc 3h sáng, retention 7 ngày. Hỗ trợ Point-in-Time Recovery (PITR) cho bất kỳ thời điểm nào trong 7 ngày.Security: RDS đặt trong private subnet của VPC, chỉ cho phép kết nối từ Security Group của ECS tasks – không có public internet access.Encryption: Storage encryption với AWS KMS key bật mặc định. d) Caching – Amazon ElastiCache RedisRedis 7 cluster mode disabled (single node) cho môi trường dev, cache.t3.micro.Đặt trong private subnet, chỉ ECS tasks có thể kết nối qua Security Group rule.NestJS sử dụng ioredis client kết nối qua Redis cluster endpoint.Cache key strategy: user:{userId}:dashboard:stats:{period} với TTL 10 phút.Cache invalidation: Khi có giao dịch mới/xóa/sửa, service tự động xóa cache key liên quan. 2.3.2. CI/CD Pipeline với AWS CodePipelineHệ thống CI/CD (Continuous Integration / Continuous Deployment) được thiết lập hoàn chỉnh với AWS CodePipeline, tự động hóa toàn bộ quy trình từ push code đến deploy production:a) Cấu trúc PipelinePipeline gồm 4 stages chính:Stage 1 – Source: Pipeline kết nối với GitHub repository, tự động trigger khi có commit mới vào branch main. Code được lưu vào S3 bucket làm artifact.Stage 2 – Build (CodeBuild): Chạy unit test (jest --coverage), build Docker image, tag với git SHA, push image lên ECR. Nếu test fail, pipeline dừng ở đây và gửi thông báo.Stage 3 – Deploy to Staging: Tự động deploy phiên bản mới lên môi trường staging trên ECS. Chạy smoke test cơ bản để verify deployment.Stage 4 – Deploy to Production (Manual Approval): Sau khi staging OK, pipeline chờ phê duyệt thủ công từ team lead. Sau khi approved, deploy lên ECS production.Hình 2.24. Biểu đồ số task đang chạy.ALB Metrics: RequestCount (số request/phút), TargetResponseTime (thời gian response trung bình), HTTPCode_Target_5XX_Count (số lỗi 5xx).RDS Metrics: DatabaseConnections (số kết nối), CPUUtilization, FreeStorageSpace, ReadIOPS/WriteIOPS.Lambda Metrics: Invocations (số lần gọi), Duration (thời gian thực thi), Errors (số lỗi).ElastiCache Metrics: CacheHits, CacheMisses, CurrConnections, BytesUsedForCache.Hình 2.25.
